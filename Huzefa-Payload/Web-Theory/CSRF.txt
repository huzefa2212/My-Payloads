CSRF ATTACK TECHNIQUES

##1 Perform the request without modifying the parameters, so we can see what the expected result is

1)Remove the CSRF token completely (in this case, the token may not always be named "csrftoken")
2)Modify one of the characters in the token (but keep the length the same)
3)Remove the value of the token (but leave the parameter in place)
4)Convert the POST request to a GET request or vice versa
5)Remove any extra parameters(other than the actual details to be sent/changed)
6)Reuse a csrf token from one account in another account for that same action
7)Look for an Auth Token before the Logging-in process(intercept the POST req)(see if this token can then be used for all users)
8)Try posting comments as other users or any other authenticated functions using your own csrf token

If any of the above steps produce the same result as ##1 then we know that the end-point is likely to be vulnerable
(there are some instances where you might get a successful response, but in fact no data has been modified and therefore the token hasn’t been checked)

Other things to lookout for +------------

1) Is the server side checking Full Length Token Based validation
2) Check the length of the CSRF token for different accounts and different functions
3) try to find the hashing algorithm, see Hashing function below ,find hash type https://crackstation.net/)
4) check if the CSRF token is simply duplicated in a cookie
5) remove the referrer header completely
