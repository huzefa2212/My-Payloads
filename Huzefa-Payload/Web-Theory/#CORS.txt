cross-origin resource sharing
-------------------------------------------------------+
-------------------------------------------------------+

~ Cross-origin resource sharing (CORS) is a browser mechanism which enables "controlled" access to resources located outside of a given domain.
~ it adds flexibility to the SOP since SOP is very restrictive
~ it prevents protection against cross-domain based attacks, however it cannot stop attacks suc as CSRF 
~ allows servers to specify who can access its assets and which HTTP request methods are allowed from external resources

Tools:
1) CORs-CLover github
2) corsy

Scenarios
1) CORS allows any origin

Request
Origin: htpps://attacker_url.com

Response header for cors
Access-Control-Allow-Origin: https://attacker_url.com
Access-Control-Allow-Credentials: true



	Access-Control-Allow-Origin: https://attacker.com
	Access-Control-Allow-Credentials: true

2) CORS allows whitelisted null value

	Access-Control-Allow-Origin: null
	Access-Control-Allow-Credentials: true

3) CORS allows any origin but not exploitable:

	Access-Control-Allow-Origin: *
	Access-Control-Allow-Credentials: true

4) CORS allows a origin which is vulnerable to XSS
	 ~ if the website is whitelisting a domain that has a XSS vulnerability, the XSS can be used to leverage an attack as if the CORS vulnerability was present in the main domain itself

5) CORS allows checks only the prefix or suffix of the Origin header eg victim.com == victimEvil.com
	~ the origin header is checked but it is still misconfigured, an attacker can create their own domain with the victims domains name in it eg if website checks for victim.com the attacker can use domain sitevictim.com
